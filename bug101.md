# Bug Investigation Report: Missing Tag Filter Row in Customer App (bug101.md)

This document contains a comprehensive analysis of the missing tag filter row on the customer category items page in the Flutter customer app.

---

## 1. Context & Expected Behavior

On the customer category page (`CategoryPage` in `mahalaxmi_customer/lib/features/category/pages/category_page.dart`):
1. The app should fetch available items for the selected category using `customerItemsByCategoryProvider`.
2. It should extract unique tags from the fetched items.
3. If there are **multiple unique tags** (`tags.length > 1`), it should display a horizontally scrollable tag filter row (`All` chip + tag chips) directly below the AppBar.
4. Selecting a chip should filter the vertical list of product cards dynamically.

---

## 2. Database & Data Analysis

We verified the live Supabase database state by writing and executing a scratch Dart test. Here are the categories, active items, and tag distributions:

*   **Total items fetched:** 29
*   **Tag distribution by category:**
    *   `Chuda Box`: 3 items, tags: `[]`
    *   `Test Cat`: 3 items, tags: `[]`
    *   `Chuda`: 11 items, tags: `['Gold', 'mehndi', 'rajwadi']` (3 unique tags)
    *   `Kaleera`: 1 item, tags: `[]`
    *   `Opek Seep Patti`: 2 items, tags: `[]`
    *   `Raw_Material`: 3 items, tags: `['Gold']` (1 unique tag)
    *   `Bhawari Chudi`: 3 items, tags: `[]`
    *   `Metal_Bangles`: 3 items, tags: `['rajwadi']` (1 unique tag)

### Key Finding
Because the tag row visibility is governed by `if (tags.length > 1)`, **the tag filter row is only expected to appear in the `Chuda` category screen.**
For all other categories (e.g., `Raw_Material`, `Metal_Bangles`), there is either `0` or `1` tag, which naturally evaluates the condition to `false` and hides the row. This matches the business rule but may look like the row has "disappeared" if viewing those categories.

## 3. UI/Layout & Code Audit

In `category_page.dart`, we audited the widget hierarchy:

```dart
body: RefreshIndicator(
  onRefresh: () {
    setState(() => _selectedTag = null);
    return ref.refresh(customerItemsByCategoryProvider(queryName).future);
  },
  child: itemsAsync.when(
    loading: () => const Center(child: CircularProgressIndicator()),
    error: (error, _) => Center(...),
    data: (items) {
      ...
      final tags = extractSortedTags(items);
      final filteredItems = filterItemsByTag(items, _selectedTag);
      return _buildContent(tags, filteredItems);
    },
  ),
)
```

And `_buildContent`:

```dart
Widget _buildContent(List<String> tags, List<RateItem> items) {
  return Column(
    children: [
      if (tags.length > 1) _buildTagRow(tags),
      Expanded(
        child: items.isEmpty
            ? _emptyFilterState()
            : ListView.separated(
                padding: const EdgeInsets.symmetric(horizontal: 16, vertical: 12),
                itemCount: items.length,
                separatorBuilder: (context, index) => const SizedBox(height: 16),
                itemBuilder: (context, index) => _ItemCard(
                  item: items[index],
                  onTap: () => context.push('/item/${Uri.encodeComponent(items[index].itemNumber)}'),
                ),
              ),
      ),
    ],
  );
}
```

### Potential Layout & Behavioral Issues

1.  **RefreshIndicator wraps the Column:**
    *   In Flutter, `RefreshIndicator` expects a scrollable child. Wrapping the entire `Column` means the refresh swipe physics must bubble up from the nested `ListView.separated`.
    *   If the list is empty (e.g., no items match the selected tag), it renders `_emptyFilterState()` (a `Center` widget with text). Because `Center` is non-scrollable, **the pull-to-refresh gesture will be completely broken** in empty states.
2.  **Tag Row layout bounds:**
    *   `_buildTagRow` returns a horizontal `ListView.separated` wrapped in a `Container` of height `52`. While this height is theoretically sufficient, horizontal lists inside column expanded structures can sometimes lose focus, overflow, or render poorly on high-density devices if not properly bounded.
3.  **State Reset during category switches:**
    *   The `_selectedTag` state is stored locally inside the stateful widget. When navigating between categories (Category A → Category B), if GoRouter reuses the same route state, `_selectedTag` might persist unless explicitly reset in `didUpdateWidget` or during navigation handlers.

---

## 4. Recommendations for the Next AI Model / Developer

To fix this cleanly:

1.  **Change the Tag Row Visibility threshold (if needed):**
    *   If you want the tag row to show even for categories with a single tag (e.g., to let users filter between "All" and "Gold"), change `if (tags.length > 1)` to `if (tags.isNotEmpty)`.
2.  **Move the RefreshIndicator Inside the Content Layout:**
    *   Place `RefreshIndicator` so it wraps only the vertical list view, leaving the tag row static at the top. This prevents the tag row from sliding down when pulling to refresh.
3.  **Fix Empty State Refreshing:**
    *   Wrap `_emptyFilterState()` inside a `ListView` or a `SingleChildScrollView(physics: AlwaysScrollableScrollPhysics())` so the pull-to-refresh swipe still works when no items are visible.
